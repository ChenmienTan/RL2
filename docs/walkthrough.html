RL2: Ray Less Reinforcement Learning

RL2 is a compact (~4.9k LOC) post-training library for LLMs — SFT, RM, DPO, and PPO/REINFORCE — built on plain torchrun, FSDP/Megatron, and SGLang. This guide walks you through how the codebase is organized, what the key abstractions are, and the order I recommend for reading the code so you can extend it confidently.

~4.9k lines of Python FSDP + Megatron backends SGLang async rollout Hydra configs, no Ray

1. What RL2 is

A "framework" but written like a script: every training step is an explicit Python statement in RL2/trainer/*.py. There is no actor-pool / engine indirection. You launch with torchrun, and the same processes own the trainer, the policy/critic models (FSDP), and they spawn local SGLang inference servers for rollout.

Design pillars

  • No Ray — pure torch.distributed + multiprocessing for SGLang.
  • Hybrid engine — same GPUs host training (FSDP) and inference (SGLang); weights are pushed at every step.
  • Sequence packing — everything is variable-length, packed into balanced minibatches.
  • Backend-pluggable — actor=fsdp or actor=megatron swaps the model engine.

What it supports

  • SFT, Reward Model, DPO, PPO, REINFORCE / Dr. GRPO / GRPO.
  • Multi-turn agentic rollout via a user-supplied env_step.
  • 3D parallelism (DP/CP/TP) on FSDP, 5D on Megatron.
  • Partial rollout, dynamic filtering, KL regularization (k1/k2/k3).

3. Core abstractions

There are really only five concepts in this library. Once you know them, every file in RL2/ reads as a variation on the same theme.

3.1 Trainer

Every training mode (sft.py, rm.py, dpo.py, ppo.py) extends Trainer. The base class only handles checkpoint resume + wandb. The subclass is the algorithm, written as a flat Python loop:

```49:86:RL2/trainer/base.py def load_ckpt(self, workers: Sequence[Worker]) -> int: if self.load_dir is None: return 0 for worker in workers: worker_name = "actor" if "Actor" in worker.__class__.__name__ else "critic" worker.load_ckpt(f"{self.load_dir}/{worker_name}/optimizer_scheduler") ckpt = self._get_ckpt(0) dcp.load(ckpt, checkpoint_id=f"{self.load_dir}/trainer") if dist.get_rank() == 0: self.train_dataloader.load_state_dict(ckpt["dataloader"]) return ckpt["step"] ```

3.2 Worker — Actor & Critic

Worker is just {config, train, tokenizer}. The interesting layer is FSDPWorker which sets up two device meshes:

  • model_device_mesh = (ddp, fsdp, tp) — used to wrap the model with hybrid-shard FSDP.
  • device_mesh = (dp, cp, tp) — used for data scattering and ring-attention along the sequence dimension.
```28:48:RL2/workers/fsdp/base.py self.model_device_mesh = dist.device_mesh.init_device_mesh( "cuda", mesh_dim_names=("ddp", "fsdp", "tp"), mesh_shape=( config.ddp_size, world_size // (config.ddp_size * config.tp_size), config.tp_size ) ) assert world_size % (config.cp_size * config.tp_size) == 0, \ f"World_size {world_size} must be divisible by cp_size {config.cp_size} * tp_size {config.tp_size}." self.device_mesh = dist.device_mesh.init_device_mesh( "cuda", mesh_dim_names=("dp", "cp", "tp"), mesh_shape=( world_size // (config.cp_size * config.tp_size), config.cp_size, config.tp_size ) ) ```

FSDPActor adds the model + the four "ops" the trainer calls: compute_logps, sft_step, dpo_step, ppo_update, plus update_rollout for weight sync. FSDPCritic uses AutoModelForTokenClassification with one label head and exposes compute_values, rm_step, ppo_update.

Mental model: the trainer never touches the model directly. It hands a tensor_dict to the worker; the worker scatters → forwards → backwards → optimizer.step → gathers, all behind one method call.

3.3 Rollout

Rollout is the only worker that doesn't hold model weights. On rank 0 it owns the dataloader + an SGLang router process; on every TP-rank-0 it spawns an SGLang server process. __call__(train, step) drives the entire rollout phase, returning a packed tensor_dict ready for training.

```67:93:RL2/workers/rollout.py class Rollout: def __init__(self, config: DictConfig): self.config = config self._prepare_device_mesh() self._prepare_environment_variables() if dist.get_rank() == 0: self.tokenizer = AutoTokenizer.from_pretrained( config.server_args.model_path, trust_remote_code=True ) self.train_dataloader, self.test_dataloader = get_dataloaders( RLDataset, config, self.tokenizer, 1 ) self._prepare_environment() self.sample_buffer: List[SampleGroup] = [] self._launch_router_process() dist.barrier(group=get_gloo_group()) if self.device_mesh["tp"].get_local_rank() == 0: self._launch_server_process() ```

3.4 Dataset / Sample / SampleGroup

SFT/RM/DPO datasets each return a tensor_dict directly. The PPO dataset is different — its items are SampleGroup objects (a group of responses_per_prompt rollouts of the same prompt) so that GRPO-style group-baseline normalization is natural.

```16:50:RL2/datasets/rl.py @dataclass class Sample: class Status(Enum): RUNNING = "running" ABORTED = "aborted" DONE = "done" # for initialization sample: Dict[str, Any] = field(default_factory=dict) # for environment interaction state_text: str = "" action_text: str = "" # for training state_dict: Dict[str, List[int | float]] = field(default_factory=dict) state_dicts: List[Dict[str, List[int | float]]] = field(default_factory=list) ```

A Sample is a small state machine: it can be RUNNING, ABORTED (used for partial rollout / preemption), or DONE. Each turn appends LLM tokens then env tokens to state_dict; if the env's next_state is not a prefix of the previous state, the sample becomes a multi-sequence trajectory.

3.5 Environment

The user contract is a single async function (or two, if you want full control of the loop):

```6:36:envs/countdown.py async def env_step(sample: Sample) -> Dict[str, Any]: env_response = { "next_state": None, "done": True, "reward": 0.0 } match = re.search(r"(.*?)", sample.action_text) if match is None: return env_response equation = match.group(1).strip() env_response["reward"] = 0.1 try: # maybe the number cannot be converted to integer numbers = [int(n) for n in re.findall(r"\d+", equation)] assert sorted(numbers) == sorted(sample.sample["numbers"]) except: return env_response try: assert re.match(r"^[\d+\-*/().\s]+$", equation) # maybe the equation is illegal result = eval(equation, {"__builtins__": None}, {}) assert abs(result - sample.sample["target"]) < 1e-5 env_response["reward"] = 1.0 except: pass return env_response generate = partial(base_generate, env_step_fn=env_step) ```

base_generate in RL2/datasets/rl.py is the prebuilt rollout loop that uses env_step. The env file just needs a module-level generate callable; the trainer points to it via rollout.env_path=envs/countdown.py.

4. Suggested reading order

If you only have an hour, read the files in this exact order. The total is well under 1,500 lines.

  1. The README — gives you the env step contract and the algorithmic recipes (Dr. GRPO default, OpenAI PPO, DeepSeek GRPO).
  2. examples/countdown_reinforce.sh — see what a real launch command looks like; every foo.bar=baz override maps 1:1 to a YAML key.
  3. RL2/trainer/ppo.py (108 lines) — the entire PPO algorithm. Read it twice.
  4. RL2/trainer/config/ppo.yaml + actor/fsdp.yaml — the schema of every hyper-parameter.
  5. RL2/workers/__init__.py — the tiny factory that picks FSDP vs Megatron.
  6. RL2/workers/rollout.py — the most novel file: SGLang lifecycle, async scheduling, and the GPU-to-GPU weight push.
  7. RL2/datasets/rl.py — Sample state machine + base_generate. This is the multi-turn engine.
  8. RL2/workers/fsdp/base.py + actor.py — model setup, scatter/gather, _forward, ppo_update.
  9. RL2/utils/algorithms.py — every loss in one ~230-line file: GAE, REINFORCE baseline, DPO, RM, PPO clip, KL k1/k2/k3.
  10. RL2/utils/sequences.py — packing/scattering/CP-sliding. Read last; it's pure plumbing.
Tip: the SFT/RM/DPO trainers (~60 lines each) are a great warm-up — they share the same Trainer base and FSDPActor/FSDPCritic classes, just without the rollout phase. If PPO feels overwhelming, read sft.py first.

5. PPO loop, line by line

This is the heart of the codebase. Everything else exists to support these ~30 lines.

6. Data & tensor flow

Every piece of training data in RL2 is eventually a Dict[str, torch.Tensor] with the same keys, regardless of whether you're doing SFT, DPO, or PPO. This uniformity is what lets the same FSDPActor._forward handle every algorithm.

The packing step (RL2/utils/sequences.py)

_tensor_dict_to_minibatches takes the flat list of variable-length sequences and uses Karmarkar–Karp bin packing to produce k balanced minibatches whose token totals are roughly equal and whose count is divisible by the number of update steps per rollout.

```17:39:RL2/utils/sequences.py """ Pack sequences into minibatches for higher throughput. There are two constrains: * The number of minibatches must be multiple of `multiple_of` * The length of any minibatch cannot exceed `max_length_per_dp` To satisfy the second constraint, the number of minibatches must be at least `math.ceil(total_length / max_length_per_dp)`. Starting from the first multiple of `multiple_of` that is no less than the value, we pack sequences into `n_minibatches` minibatches and check whether the second constraint is satisfied. If not, we increase `n_minibatches` by `multiple_of` (so that the first constraint is always satisfied) and repeat the loop. """ seq_len_list = (tensor_dict["eos_mask"].argmax(-1) + 1).tolist() ```
Gotcha: scatter_data uses module-level globals (SHUFFLE_INDICES, PAD_SEQUENCES) so the matching gather_data can undo the partition. This means you can't have two scatter/gather calls in flight on different threads.

7. Parallelism (DDP / FSDP / CP / TP)

RL2 uses a single torchrun world and slices it three ways for the training engine and a separate way for the rollout engine.

Training device mesh

Built in FSDPWorker.__init__. Two views over the same processes:

  • (ddp, fsdp, tp) — feeds FSDP(sharding=HYBRID_SHARD): shard inside the fsdp dim, replicate across ddp.
  • (dp, cp, tp) — used to scatter data and to drive ring-flash-attention along cp.

Rollout device mesh

Built in Rollout._prepare_device_mesh. Just (dp, tp) on CPU — only used to know who launches an SGLang server (TP rank 0) and who routes (rank 0 globally).

FSDP wrap policy

```44:53:RL2/utils/fsdp/data_parallelism.py return FSDP( model, auto_wrap_policy=auto_wrap_policy, sharding_strategy=ShardingStrategy.HYBRID_SHARD, mixed_precision=mixed_precision, param_init_fn=_param_init_fn, sync_module_states=sync_module_states, device_mesh=device_mesh, device_id=torch.cuda.current_device() ) ```

Context parallelism — Zigzag ring attention

Sequences are split along the time dimension across cp_size ranks using a zigzag pattern (chunk i and chunk 2c−i−1 on rank i) so each rank carries a balanced amount of causal-attention work. The attention forward is replaced by zigzag_ring_flash_attn_varlen_func via a global monkeypatch in RL2/utils/fsdp/context_parallelism.py.

Tensor parallelism + logits

When TP > 1, each rank holds a vocab shard. Computing log-probs requires (a) chunked logsumexp across the local vocab + an all_gather+logsumexp across ranks, and (b) a "gather your own actions" pattern with a differentiable all_reduce. See compute_logps_and_entropy in RL2/utils/functions.py.

Algorithm dial: the same actor_ppo_loss + compute_advantages implements REINFORCE, Dr. GRPO (default), GRPO, and OpenAI PPO. Switching between them is a matter of toggling adv.estimator, kl.type, actor.avg_level, and adv.norm_var. Read the README "Algorithm" section for the exact dial settings.

9. Hybrid engine: weight sync to SGLang

The single most distinctive piece of the codebase. After every PPO update, the FSDP-sharded actor pushes its weights to the local SGLang server without going through the network — using shared-memory tensor handles.

flowchart LR A["FSDPActor.update_rollout
(yields named DTensor params)"] --> B["Rollout.update
(buckets by dtype)"] B --> C["DTensor.redistribute([Replicate])
→ local full tensor on GPU"] C --> D["FlattenedTensorBucket
+ MultiprocessingSerializer.serialize()"] D --> E["dist.gather_object across TP
→ TP-rank-0"] E --> F["HTTP POST /update_weights_from_tensor
(sends only metadata; actual
tensor copied via shared mem)"] F --> G[SGLang server applies weights]

Two HTTP endpoints from SGLang make this work:

  • release_memory_occupation — called after rollout to free KV cache + weights.
  • resume_memory_occupation — called before weight update; only weights are resumed first to avoid OOM, then KV cache is resumed after the new weights are installed.

Why it's fast: MultiprocessingSerializer + FlattenedTensorBucket exchange CUDA IPC handles, so the SGLang process maps the actor's GPU memory directly. The HTTP body contains only metadata.

10. What you can build with it

Concretely, here's what each trainer enables out of the box.

Supervised Fine-Tuning

Standard cross-entropy on prompt/response or multi-turn messages. Both formats supported by SFTDataset._tokenize_prompt_response / _tokenize_messages.

Reward Modeling

Pairwise Bradley–Terry on chosen/rejected, packing chosen+rejected together (token-classification head, single scalar per token, masked to EOS).

DPO

Pairwise sigmoid loss using logp − ref_logp. Reference policy is a frozen actor; ref logps are computed once per batch.

PPO / REINFORCE / GRPO / Dr. GRPO

Multi-turn agentic rollout with per-token rewards, optional value head (GAE) or group baseline (REINFORCE family). Configurable KL placement (reward vs loss) and estimator (k1/k2/k3).

Multi-turn agents

Any environment expressible as async env_step(sample) -> {next_state, reward, done}. Examples: tool-use search (SearchR1), countdown reasoning, deep research, GEM environments.

Eval-only

Set trainer.eval_only=true and a single rollout(False, 0) runs your env across the test split with greedy decoding.

11. Extending RL2

Add a new environment

  1. Create envs/my_env.py with an async def env_step(sample) -> dict.
  2. At module top-level, expose generate = partial(base_generate, env_step_fn=env_step).
  3. Set rollout.env_path=envs/my_env.py on the launch command.
  4. Make sure your dataset rows have whatever env_step reads from sample.sample (e.g. extra_info.answer).

Add a new advantage / loss

  1. Add a function in RL2/utils/algorithms.py alongside _compute_gae.
  2. Wire it into the dispatcher inside compute_advantages (~line 127).
  3. Add the corresponding YAML keys to RL2/trainer/config/ppo.yaml under adv:.

Add a new training mode

  1. Subclass Trainer in RL2/trainer/my_mode.py.
  2. Add a corresponding YAML in RL2/trainer/config/ with a defaults: list pointing at actor: fsdp.
  3. Add a step-style method on FSDPActor / FSDPCritic if the loss doesn't fit existing ones.
  4. Launch with torchrun -m RL2.trainer.my_mode.

Add a new model backend

Look at the existing pattern in RL2/workers/__init__.py: each backend folder (fsdp/, megatron/) defines {Backend}Actor and {Backend}Critic with the same method surface (compute_logps, ppo_update, update_rollout, …). Add a my_backend/ folder, register it in the factory, and add a YAML in RL2/trainer/config/actor/.
